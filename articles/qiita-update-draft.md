## 1. 「自然言語ファースト」への転換

### 前回の課題

前回の記事では `/screen-stocks japan value --top 10` のようなコマンド体系を紹介した。動くのだが、ユーザー体験として「コマンドを覚える必要がある」という本質的な課題があった。

Claude Code Skills の強みは、LLM がインターフェースそのものであること。であれば、ユーザーは日本語で意図を伝えるだけでよいはずだ。

### Intent Routing：2段階ドメインルーティング

`.claude/rules/intent-routing.md` に「自然言語 → スキル判定」のルールを定義した。

**Step 1: ドメイン判定** — ユーザーの発言を8つのドメインに分類する。

| ドメイン | 代表的な表現 | 対応スキル |
|:---|:---|:---|
| 発見 | 「いい日本株ある？」「高配当株を探して」 | `/screen-stocks` |
| 分析 | 「トヨタってどう？」「還元率は？」 | `/stock-report` or `/market-research` |
| 保有管理 | 「PF見せて」「100株買った」 | `/stock-portfolio` |
| リスク | 「暴落したらどうなる？」 | `/stress-test` |
| 監視 | 「気になるから記録して」 | `/watchlist` |
| 記録 | 「メモしておいて」「学びを記録」 | `/investment-note` |
| 知識 | 「前回のレポートは？」「常連銘柄は？」 | `/graph-query` |
| メタ | 「何ができる？」「改善点ある？」 | システム自己分析 |

**Step 2: ドメイン内でスキル・パラメータを推定。**

```
ユーザー: 「Xで話題のAI株を探して」
→ ドメイン: 発見
→ スキル: /screen-stocks
→ パラメータ: --region us --preset trending --theme "AI"
```

### ポイント：ルールファイルは「LLM への指示書」

`intent-routing.md` は Python コードではない。Claude が読んで判断するためのルールドキュメントだ。キーワードマッチではなく、文脈から意図を推論する。

```markdown
# intent-routing.md の一部
## 原則
1. **ユーザーの意図を最優先で汲み取る** — キーワードマッチではなく、文脈から意図を推論する
2. **曖昧なときは確認する** — 複数のスキルが該当しそうな場合、短く選択肢を提示する
3. **会話の流れを引き継ぐ** — 直前に扱った銘柄やPFの状態を覚えて、省略された情報を補完する
```

これにより、以下のような会話の流れが自然に成立する：

```
ユーザー: 「トヨタってどう？」
→ /stock-report 7203.T を実行

ユーザー: 「ウォッチリストに入れて」
→ 直前のコンテキストから 7203.T を補完 → /watchlist add 7203.T

ユーザー: 「もっと詳しく」
→ /market-research stock 7203.T に連鎖

ユーザー: 「PFに追加したらどう変わる？」
→ /stock-portfolio what-if --add "7203.T:100:2850"
```

## 2. 新スキル：深掘りリサーチ（market-research）

### 課題：yfinance だけでは足りない

`/stock-report` は yfinance の財務データに基づく定量分析に強い。しかし「最新ニュース」「Xでの評判」「業界トレンド」といった定性情報はカバーできない。

### Grok API 統合

xAI の Grok API を統合し、4種類の深掘りリサーチを実現した。

| タイプ | 対象 | 例 |
|:---|:---|:---|
| `stock` | 個別銘柄 | 「NVDAの最新情報」 |
| `industry` | 業界 | 「半導体業界を調べて」 |
| `market` | マーケット全体 | 「今の相場は？」 |
| `business` | ビジネスモデル | 「トヨタは何で稼いでる？」 |

Grok API は X（旧Twitter）のリアルタイムデータと Web 検索結果の両方にアクセスできる。これにより「機関投資家の間でどう語られているか」「直近のカタリストは何か」といった定性情報を取得できるようになった。

### Graceful Degradation

`XAI_API_KEY` が未設定でもシステムは動作する。Grok 連携部分がスキップされ、yfinance ベースの分析のみが実行される。

```python
try:
    from src.data import grok_client
    HAS_GROK = True
except ImportError:
    HAS_GROK = False
```

この `HAS_MODULE` パターンはシステム全体で一貫して使っている設計方針だ。

## 3. 新スキル：投資メモ（investment-note）

### 動機：「なぜ買ったか」を忘れる問題

投資で最も重要なのは「買った理由」と「売った理由」の記録だ。しかし多くの個人投資家はこれを体系的に残さない。

### 6タイプの投資メモ

| タイプ | 用途 | 例 |
|:---|:---|:---|
| `thesis` | 投資テーゼ | 「EV普及で部品需要増」 |
| `observation` | 気づき・観察 | 「決算で海外売上比率が上昇」 |
| `concern` | 懸念・リスク | 「中国依存度が高い」 |
| `review` | 振り返り | 「半年持ったが想定通り推移」 |
| `target` | 目標株価 | 「3500円で利確検討」 |
| `lesson` | 学び・教訓 | 「決算跨ぎは避けるべきだった」 |

### 売買との自然な連携

Intent Routing の「複合意図の自動連鎖」により、売買とメモが自然につながる。

```
ユーザー: 「トヨタ100株買った、理由もメモして」
→ 1. /stock-portfolio buy --symbol 7203.T --shares 100 --price ...
→ 2. /investment-note save --symbol 7203.T --type thesis --content ...
```

## 4. Neo4j ナレッジグラフ：「記憶」の獲得

### 前回の最大の課題：コンテキスト消失

前回の記事で最も大きな課題として挙げたのが「コンテキスト消失」だった。

> Web版ChatGPTやClaudeでは、セッションが終わるとそれまでの分析結果が失われる。

Claude Code Skills でも、会話セッション間でのコンテキストは保持されない。JSON ファイルに履歴を書き出してはいたが、「過去に何を分析したか」を構造的に検索することはできなかった。

### Neo4j の導入

この問題を解決するために Neo4j ナレッジグラフを導入した。

#### 19ノード × リレーション

```
Stock ← SURFACED ← Screen       （スクリーニングで浮上）
Stock ← ANALYZED ← Report       （レポートで分析）
Stock ← BOUGHT/SOLD ← Trade     （売買記録）
Stock ← HOLDS ← Portfolio       （現在保有）
Stock ← BOOKMARKED ← Watchlist  （ウォッチ中）
Stock ← ABOUT ← Note            （投資メモ）
Research → HAS_NEWS → News      （ニュース）
Research → HAS_SENTIMENT → Sentiment  （センチメント）
Research → SUPERSEDES → Research      （新旧チェーン）
```

#### Dual-Write パターン

**JSON がマスター、Neo4j はビュー。** スキル実行時に JSON/CSV への書き込みが常に先に行われ、Neo4j への同期はベストエフォートで行う。

```python
# history_store.py の書き込みフロー
def save_report(symbol, data):
    # 1. JSON 書き込み（マスター）— 必ず成功
    _write_json("report", symbol, data)

    # 2. Neo4j 同期（ビュー）— 失敗しても OK
    try:
        graph_store.save_report_node(symbol, data)
    except Exception:
        pass  # graceful degradation
```

これにより、Neo4j が落ちていてもシステムは正常に動作し続ける。

#### NEO4J_MODE：書き込み深度の制御

Neo4j への書き込み量を3段階で制御できる。

| モード | 書き込み内容 | 用途 |
|:---|:---|:---|
| `off` | 書き込みなし | Neo4j 不要時 |
| `summary` | スコア・判定・サマリーのみ | 軽量運用 |
| `full` | ニュース・センチメント・カタリスト等も展開 | フル活用 |

### 新スキル：知識グラフ検索（graph-query）

蓄積された知識を自然言語で検索できる。

```
「7203.Tの前回レポートは？」
→ 直近のレポートノードを取得、スコア・判定を表示

「繰り返し候補に上がってる銘柄は？」
→ 複数回 SURFACED された銘柄を集計

「AAPLのセンチメント推移」
→ 時系列の Sentiment ノードを取得
```

内部実装は `graph_nl_query.py` のテンプレートマッチ → `graph_query.py` の Cypher クエリ実行という2段構成。

## 5. 自動コンテキスト注入（KIK-411）

### 「トヨタってどう？」の裏側

ユーザーが「トヨタってどう？」と聞いたとき、このシステムは `/stock-report` を実行する前に、まず過去の経緯を自動取得する。

```bash
python3 scripts/get_context.py "トヨタってどう？"
```

このスクリプトが行うこと：

1. **シンボル検出**: 「トヨタ」→ `7203.T` に変換
2. **Neo4j 照会**: 過去のレポート・売買・ウォッチリスト・メモを検索
3. **関係性判定**: 保有中？ウォッチ中？初見？
4. **スキル推奨**: 関係性に応じた最適なスキルを提案

例えば、7203.T を保有中なら「ヘルスチェック優先」、ウォッチ中なら「前回との差分を意識したレポート」、初見なら「通常のレポート」と推奨が変わる。

### グラフ状態によるスキル推奨

| 関係性 | 推奨スキル | 理由 |
|:---|:---|:---|
| 保有銘柄 | health（ヘルスチェック） | 保有者として診断優先 |
| テーゼ3ヶ月経過 | health + レビュー促し | 定期振り返りタイミング |
| ウォッチ中 | report + 前回差分 | 買い時かの判断材料 |
| 3回以上スクリーニング出現 | report + 注目フラグ | 繰り返し上位で注目度高 |
| 未知の銘柄 | report | ゼロから調査 |

## 6. スクリーニングの進化

### 新プリセット

前回の7プリセットに3つを追加した。

#### trending：X トレンド銘柄

Grok API で X（旧Twitter）のトレンドを取得し、ファンダメンタルズフィルタと組み合わせる。

```
ユーザー: 「Xで話題の株は？」
→ Grok API で話題銘柄を取得
→ PER/PBR/ROE等でフィルタ
→ ファンダメンタルズが裏付ける銘柄のみ表示
```

「バズっているだけ」の銘柄を排除し、ファンダメンタルズの裏付けがある銘柄だけを残すのがポイント。

#### long-term：長期投資候補

ROE≥15%、EPS成長≥10%、配当≥2%、時価総額上位の安定大型株をスクリーニング。

#### shareholder-return：株主還元

配当 + 自社株買いの**総還元率**でスクリーニング。さらに還元の安定度を4段階で評価する。

| マーク | 意味 | 判定基準 |
|:---|:---|:---|
| ✅ | 安定還元 | 3年間安定的に高還元 |
| 📈 | 増加傾向 | 還元率が上昇トレンド |
| ⚠️ | 一時的高還元 | 特別配当・大型自社株買いの可能性 |
| 📉 | 低下傾向 | 還元率が下降トレンド |

## 7. ポートフォリオ管理の進化

前回の6サブコマンドから12サブコマンドに拡張した。新機能をいくつか紹介する。

### ヘルスチェックの強化

#### バリュートラップ検出

「PER が低いから割安」は危険な思い込みだ。低PER + 利益減少の組み合わせは、割安ではなく「業績悪化を織り込んだ適正価格」の可能性がある。

```
⚠️ バリュートラップ警告: 7203.T
  PER: 8.5（低い）
  EPS成長率: -15%（減少）
  → 低PERは業績悪化の反映の可能性あり
```

#### ゴールデンクロス / デッドクロス検出

60日間のルックバックで SMA クロスイベントを検出し、テクニカルシグナルとして表示する。

#### 還元安定度評価

ヘルスチェックで各保有銘柄の株主還元安定度を評価。一時的高還元（⚠️）は早期警告に昇格し、長期適性判定にも総還元率を使用する。

### What-If シミュレーション

「この銘柄を追加したらPFはどう変わるか？」を事前にシミュレーションできる。

```
ユーザー: 「NVDAを10株追加したらどうなる？」
→ セクター構成の変化、集中度(HHI)の変化、リスク指標の変化を表示
```

### リバランス提案

HHI（ハーフィンダール・ハーシュマン指数）ベースの集中度分析を行い、リスク制約を満たすリバランス案を提案する。

### 複利シミュレーション

3シナリオ（楽観・基本・悲観）× 配当再投資 × 月次積立で、将来の資産推移をシミュレーションする。

```
ユーザー: 「月5万追加して5年後に1500万いける？」
→ /stock-portfolio simulate --years 5 --monthly-add 50000 --target 15000000
```

## 9. 開発フローの進化

### Worktree ベースの開発

前回は `git branch` ベースだったが、現在は **Worktree ベース**に移行した。

```bash
# Linear issue KIK-413 に対して
git worktree add -b feature/kik-413-neo4j-full-mode ~/stock-skills-kik413 main
```

main ブランチに触れないため、常に動作する状態が保たれる。

### コードレビュー：エージェントチーム

実装後にエージェントチームを組んでコードレビューを行う。

| レビュアー | 観点 |
|:---|:---|
| arch-reviewer | 設計・構造・責務分離 |
| logic-reviewer | ロジック・エッジケース・異常値ガード |
| test-reviewer | テストカバレッジ・境界値・モック適切さ |

3エージェントが並列にレビューし、指摘があれば修正→再テスト→再レビューのサイクルを回す。

### 結合試験：並列テスト

コードレビュー後、4エージェントが並列に結合試験を実行する。

| テスター | 担当 |
|:---|:---|
| screener-tester | スクリーニング（複数プリセット・リージョン） |
| report-tester | レポート + ウォッチリスト |
| portfolio-tester | ポートフォリオ全サブコマンド |
| stress-tester | ストレステスト複数シナリオ |

### ドキュメント更新の自動チェック

`src/core/` や `src/data/` を変更した際にドキュメント更新を忘れると、3段階で検知される：

1. **PostToolUse hook**: 編集時にリマインド表示
2. **Stop hook**: 会話終了時に未更新を指摘
3. **pre-commit hook**: ドキュメント未更新の commit をブロック

## 10. 設計で得た追加の知見

### Dual-Write パターンの有効性

「JSON がマスター、Neo4j はビュー」という設計は、個人開発において非常にうまく機能した。

- Neo4j が落ちていても JSON は必ず書ける → **システムは常に動く**
- JSON から Neo4j を再構築できる → **データ損失リスクがない**
- Neo4j の MERGE 操作で冪等性を確保 → **同じデータを何度書いても安全**

### Rules ファイルによるLLM行動制御

`.claude/rules/` ディレクトリに置いたルールファイルは、Claude Code が自動的に読み込んで行動指針にする。コードではなく、自然言語でLLMの振る舞いを定義できる。

現在のルールファイル：

| ファイル | 役割 |
|:---|:---|
| `intent-routing.md` | 自然言語→スキル判定 |
| `graph-context.md` | 自動コンテキスト注入 |
| `workflow.md` | 開発ワークフロー |
| `development.md` | コーディング規約 |
| `screening.md` | スクリーニング開発ルール |
| `portfolio.md` | ポートフォリオ開発ルール |
| `testing.md` | テスト開発ルール |

### SUPERSEDES チェーン：データの新旧管理

同じ銘柄に対するレポートやリサーチは、`SUPERSEDES` リレーションでチェーンされる。「前回のレポートは？」と聞くと、このチェーンを辿って最新と直前のデータを比較表示できる。

## まとめ

前回は「Claude Code Skills で投資分析を自動化できる」ことを示した。今回は、そこから一歩進んで以下を実現した。

1. **自然言語ファースト**: コマンドを覚える必要がない。日本語で話しかけるだけ
2. **ナレッジグラフによる記憶**: 過去の分析結果が構造化されて蓄積される
3. **自動コンテキスト注入**: 銘柄名を言うだけで、過去の経緯が自動的に引き出される
4. **エージェントチームによる品質保証**: レビュー3名 + テスト4名の並列実行
5. **Graceful Degradation の徹底**: Neo4j が落ちても、Grok API がなくても、動く

特に3つ目の「自動コンテキスト注入」が体験上の最大の変化だ。保有銘柄について聞けば「あなたは3ヶ月前にこのテーゼで買っている」という文脈が自動的に付与される。これは Web 版の AI ツールでは実現しにくい、ローカル実行ならではの強みだと考えている。
